### Kararlar (K1–K11)

**K1 — Kimlik: "yazıcı" = "agent". Ayrı `printers` tablosu AÇILMAZ.** ADR-004 §5 gereği model 1 agent = 1 yazıcı. Bugün agent'tan bağımsız bir yazıcı kaydı yok; ayrı tablo 1:1 boş-join + senkronize tutulacak ikinci kimlik demek = v3'ün `type`+`roles` ölü-paralel-kavram hatası. `agents` tablosu **iki nullable kolonla** genişletilir: `display_name TEXT` (admin'in yazdığı Türkçe etiket, ör. "Mutfak Yazıcısı (JP80H)"; NULL ise UI `device_fingerprint`'e düşer) ve `declared_kinds TEXT[]` (K2). **Fiziksel ayarlar (transport/port/spooler adı/codepage) buluta KOPYALANMAZ** — tek kaynak dükkan PC'sindeki config dosyasıdır; bulutta bir kopyası olsaydı yalan olurdu (bulut onu ne okur ne yazar). UI bunu açıkça söyler: "Fiziksel ayarlar bu yazıcının kurulu olduğu PC'deki config dosyasındadır." **UI'da "agent" kelimesi geçmez; kullanıcıya "yazıcı" denir** (glossary'ye eklenir).

**K2 — `declared_kinds` = GÖZLENEN, otoriter DEĞİL.** Claim endpoint'i (`GET /print/v1/jobs/next`) agent'ın zaten gönderdiği `?kind=` dizisini normalize ettikten sonra `agents.declared_kinds`'a **fire-and-forget** yazar (`last_seen_at` ile birebir aynı desen, aynı satırda). Sunucu bu alanı **claim filtresinde KULLANMAZ** — ADR-032 Design B (agent kendi filtresini beyan eder) canlı mekanizma olarak aynen kalır. Gerekçe: (a) **dürüstlük** — ekran, agent'ın gerçekte ne çektiğini gösterir, temenniyi değil (v3 `roles` hatasının panzehiri); (b) claim hot-path'inin davranışı **bit-bit aynı** kalır → cutover riski sıfır; (c) sunucu-otoriter rol (ADR-032 Design A) v5.1'de bu veriyi hazır bulur. `?kind=` göndermeyen agent → `NULL` → UI "Tüm işleri alıyor (filtresiz)" **uyarı çipi** basar; bu, ADR-032'nin (−) maddesindeki en tehlikeli yanlış-config'in ilk kez görünür olmasıdır.

**K3 — Kategori→yazıcı ataması: `categories.kitchen_print` TEK OTORİTE olarak KALIR; UI onu yazıcı-seçici olarak sunar. `[ONAY GEREKLİ]`** Yeni `category_printers` join tablosu **v5.0'da açılmaz**. Gerekçe üç katmanlı: (1) `kind` kümesi donduruldu ve bugün **tek** mutfak yazıcısı var → routing tablosu **sıfır ek ifade gücü** verir, saf karmaşıklıktır; (2) join tablosu **iki hot-path'in birden** (`orders.ts:641` + `kds.ts:145`) yeniden yazılmasını gerektirir — cutover'a 4 gün kala kabul edilemez; (3) `kitchen_print` NOT NULL DEFAULT TRUE boolean olduğundan **"atanmamış kategori" durumu yapısal olarak yoktur** → v3'ün tip-güvensiz 4-kademeli fallback'i v5'te **imkânsızdır** (kural: doğru tipte yazıcı yoksa iş kuyrukta bekler, ASLA yanlış yazıcıya düşmez — ADR-032 kenar durum 3). **Bir kategori birden çok yazıcıya atanamaz** (v5.0): tek mutfak hattı + v3'ün "UNIQUE'i sonradan yamamak" dersi → ilişki gün-1'den 1:1. UI'da kategori satırındaki seçim **radyo listesi**dir: kayıtlı `kitchen`-kind yazıcılar + "Yazdırma (mutfağa düşmesin)". **Onay modali iki sonucu birden yazar:** "Bu kategorinin ürünleri artık mutfak yazıcısından basılmayacak **ve mutfak ekranında (KDS) görünmeyecek."** v5.1'de 3. hat gelince depolama join tablosuna taşınır; UI sözleşmesi (seçici) aynı kalır.

**K4 — "Yazıcı ekleme" = dürüst 2 aşamalı akış (bulut + PC).** Bulut, restoran PC'sine Windows servisi kuramaz; UI bunu iddia etmez. Akış: **(1)** Admin "Yazıcı Ekle" → form: görünen ad + iş türü (`Mutfak`/`Kasa`) + cihaz kimliği (`device_fingerprint`, ör. `mutfak-2`). **(2)** Sunucu `agents` satırını **bekleyen (pending)** durumda oluşturur — `generateAgentApiKey`/`hashAgentApiKey` mevcut helper'larıyla anahtar üretilir, **yalnız bcrypt hash saklanır**; ham anahtar yanıt gövdesinde **BİR KEZ** döner ve ekranda kopyala-butonlu, "Bu anahtar bir daha gösterilmeyecek" uyarısıyla sunulur. **(3)** Ekran, dükkan PC'sinde çalıştırılacak **hazır PowerShell komutunu** üretir (`install-second-agent.ps1` parametreleri önden doldurulmuş: `-ServiceName -ConfigPath -JobKinds -PrinterName -DeviceFingerprint -ApiUrl`, `-SetApiKey` interaktif kalır) + kopyala butonu + açık etiket: "Bu komut **dükkan PC'sinde** çalıştırılır. Bulut bu kurulumu uzaktan yapamaz." **(4)** Agent kendini **mevcut** `POST /print/v1/agent/register` ile yazar; satır ilk `last_seen_at` ile *bekleyen* → *çevrimiçi* geçer. **Register kontratı DEĞİŞMEZ.** İmplementer doğrulaması (DoD): önceden oluşturulmuş satır + aynı fingerprint, register'ın **idempotent** dalına düşmeli, `409 AGENT_FINGERPRINT_CONFLICT`'e değil; düşmüyorsa **bekleyen-satır şekli** düzeltilir, register kontratı DEĞİL.

**K5 — "Silme" = devre dışı bırakma (revoke) + gerçek engel + PC talimatı.** Hard delete **yok** (denetim izi + geçmiş işler). `revoked_at` set etmek servisi durdurmaz; yalnız agent'ın sonraki isteğini `print-agent-auth.ts:105-113` üzerinden 401'e düşürür — modal bunu **açıkça** yazar ve PC'de yapılacak işi verir: `install-second-agent.ps1 -Uninstall -ServiceName <ad>` (yoksa servis 401 alarak dönmeye devam eder). Modal, v3'ün yalanını tekrarlamaz: **gerçek kullanım özeti** (bu yazıcıya yönlenen kategori sayısı · kendi `kind`'ında bekleyen/başarısız iş sayısı · son görülme) + **gerçekten uygulanan blokerler**: (a) bir `kind`'ı üstlenen **son çevrimiçi yazıcı** devre dışı bırakılamaz (o hattın baskısı sessizce durur) — engel metni sayıyla; (b) o `kind`'da bekleyen iş varsa ve devralacak başka aktif yazıcı yoksa engel. Engel varsa buton **gerçekten pasif** olur. `revoke_reason` (mevcut kolon) zorunlu serbest metin. **Geri alma (`restore`) UI'da vardır** — yanlış tıklama SQL gerektirmeden düzeltilebilmeli.

**K6 — Test baskısı: v3 dersi aynen uygulanır (kuyruktan geçer).** `POST /printers/:id/test-print` doğrudan yazıcıya yazmaz; **normal `print_jobs` kuyruğuna** bir satır atar, `payload.kind` = o yazıcının beyan ettiği kind → mevcut ADR-032 filtresi işi doğru agent'a teslim eder. İçerik **ADR-004 Amd9 raster boru hattıyla** üretilir (yeni küçük şablon `test-receipt.ts`: işletme adı + yazıcı görünen adı + cihaz kimliği + tarih/saat + Türkçe alfabe + `₺` + buzzer) → tek tıkla **kuyruk→claim→transport→ESC/POS→kağıt** zincirinin tamamı ve Türkçe/₺/bip birlikte sınanır. UI işin sonucunu `GET /printers/jobs/:jobId` ile yoklar ve `success`/`failed` + hata metnini gösterir. **Dürüstlük notu (v5.0 sınırı):** `print_jobs`'ta `printer_id` yok → aynı `kind`'ı beyan eden iki yazıcı olsaydı testin hangisine düşeceği garanti edilemezdi; bugünkü 2-yazıcı kurulumunda (1 kitchen + 1 bill) belirlenimlidir. Bu sınır ekranda dipnot olarak yazılır.

**K7 — Durum/monitoring eşikleri.** `GET /printers` her yazıcı için: görünen ad · cihaz kimliği · `declared_kinds` · `last_seen_at` · `revoked_at` · hesaplanan durum · kendi kind'ındaki kuyruk derinliği. Agent ≤25 sn long-poll yaptığından: **Çevrimiçi** `last_seen_at < 60 sn` (2× en kötü long-poll + pay) · **Gecikmeli** 60 sn – 5 dk · **Çevrimdışı** > 5 dk veya hiç görülmemiş · **Devre dışı** `revoked_at IS NOT NULL` · **Bekliyor** (kaydedildi, hiç register olmadı). **"Sorunlu" tanımı** (v3'ün kırmızı sayacının v5 karşılığı) — şu dördünden biri: çevrimdışı · kendi kind'ında `failed`/`retry` iş var · kind beyanı yok (filtresiz çekiyor) · **bir `kind`'ı üstlenen çevrimiçi yazıcı yok = yetim kuyruk**. Sonuncusu bu ekranın en yüksek operasyonel değeri: ADR-032'nin yazılı riski ("iki agent de `["bill"]` → mutfak hiç basmaz") ilk kez görünür olur.
