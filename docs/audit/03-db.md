## 0. Yönetici özeti

**En önemli iki sonuç zıt yönde:**

**✅ BLOCKER ekseni (cross-tenant) TEMİZ.** Serinin ana korkusu — bir repo fonksiyonunun `tenant_id` filtresini unutması — **gerçekleşmemiş.** ~120 exported fonksiyon / 103 mutation-query tek tek denetlendi; hepsi tenant-scoped veya bilinçli-global (dokümante: refresh-token hash=bearer, retention cron'ları). SQL injection **yok** (tüm raw `sql\`\`` template'leri parametre-bind). RLS ADR-003 §6.4 ile v5.2'ye ertelendiğinden tek savunma manuel repo-filtresi — ve disiplin eksiksiz. **12/12 canlı cross-tenant testi yeşil.**

**🔴 Ama tx-atomikliğinde 2 canlı-kanıtlı BLOCKER var** (para-kritik, prod `restoranpos.org` canlı):
1. **DB-TX-05** — ödeme idempotency-race recovery'si PostgreSQL'in "aborted transaction" kuralı yüzünden HİÇ çalışmıyor → çift-ödeme koruması pratikte devre dışı.
2. **DB-TX-01** — `addItems`/`updateItemTx` sipariş satırını `FOR UPDATE` ile kilitlemiyor (kardeş `payOrderTx`/`cancelOrder` kilitliyor) → eşzamanlı iptal+kalem-ekleme "cancelled ama total>0 + aktif kalem" tutarsız state üretiyor.

İkisi de küçük fix (SAVEPOINT + `.forUpdate()`) ama kök çözüm repo+route birlikte → **Blok 5'e (orders/payments) bağlanır.** Migration serisi olgun (0 BLOCKER/HIGH — ekip kendi hatalarını sonraki migration'larda kapatmış: 043←027/032, 042←041, 028←026).

### En kritik 3 bulgu
1. **DB-TX-05** (BLOCKER) — idempotency-race aborted-tx: çift ödeme riski. `SAVEPOINT` ile 1-fonksiyon fix.
2. **DB-TX-01** (BLOCKER) — addItems kilitsiz: iptal+ekleme tutarsızlığı. `.forUpdate()` ile 2-satır fix.
3. **DB-SEC-01** (HIGH/KVKK) — `mapPgError` unique-detail'i `.message`'a koyuyor → e-posta/telefon log'a sızabilir.

### 2.1 BLOCKER (2) — ikisi de canlı pos_test'te kanıtlandı

### [BLOCKER] [BUG] Ödeme idempotency-race recovery'si aborted-transaction yüzünden hiç çalışmıyor (ID: DB-TX-05)
- **Dosya:** `packages/db/src/repositories/payments.ts:265-284` (özellikle :270)
- **Kanıt (ana-context + canlı doğrulandı):** INSERT (`:245-264`) idempotency_key ile; unique-violation (23505) catch'inde replay SELECT (`:270`) **aynı `trx` içinde**. PG kuralı: tx içinde herhangi bir hata (SAVEPOINT yoksa) tx'i "aborted" yapar → sonraki her komut `25P02 current transaction is aborted`. Hat C iki-client interleave ile gerçek `25P02` üretti (`payment-idempotency-race.findings.test.ts`, KIRMIZI).
- **Senaryo:** İki eşzamanlı `POST /payments` aynı idempotencyKey ile (çift-tıklama "Öde" / timeout-retry). Kaybeden 23505 alır → replay SELECT 25P02 ile patlar (mapPgError 25P02'yi de tanımaz → DB-TX-03) → kasiyer anlaşılmaz hata görür, tekrar dener; client yeni key üretirse **gerçek çift ödeme kaydı**.
- **Etki:** İdempotency'nin tüm amacı (çift ödeme önleme) bu satırda devre dışı — POS'ta finansal bütünlük.
- **Öneri:** INSERT'i `SAVEPOINT` ile sar; catch'te `ROLLBACK TO SAVEPOINT` sonra replay SELECT. Alternatif: replay'i taze bağlantıda.
- **Etiket:** MVP-fix (BLOCKER — kök çözüm Blok 5 orders/payments ile birlikte; fix onay bekliyor)

### [BLOCKER] [BUG] `addItems`/`updateItemTx` sipariş satırını kilitlemiyor — iptal edilmiş siparişe kalem eklenebiliyor (ID: DB-TX-01)
- **Dosya:** `packages/db/src/repositories/orders.ts:702-707` (addItems), `:790-795` (updateItemTx) — kilitsiz SELECT
- **Kanıt (ana-context + canlı doğrulandı):** Grep: `payOrderTx`(:899), `cancelOrder`(:962), `assignCustomer`(:1323), `moveToTable`(:1382), `mergeInto`(:1536) hepsi `.forUpdate()`; **addItems/updateItemTx kullanmıyor** (asimetri). Hat C interleave: addItems kilitsiz SELECT 'open' okur → gerçek `cancelOrder` commit eder (cancelled, total=0) → addItems yazma fazı stale-read üzerinden devam → sonuç `status='cancelled' ama total_cents=22000 + 1 aktif kalem` (`addItems-missing-lock.findings.test.ts`, KIRMIZI). **DB'de de backstop yok** (migration trigger'ları order_items INSERT'ini orders.status'a bağlamıyor).
- **Senaryo:** Garson kalem eklerken kasiyer aynı anda iptal ederse → iptal edilmiş siparişte aktif kalem + pozitif tutar; mutfağa "iptal" görünüp fişte tutar çıkabilir.
- **Öneri:** addItems/updateItemTx order SELECT'ine `.forUpdate()` ekle (kardeş fonksiyon deseni).
- **Etiket:** MVP-fix (BLOCKER — Blok 5 ile; fix onay bekliyor)

### 2.2 HIGH (5)

### [HIGH] [SEC/KVKK] `mapPgError` unique-detail'i `.message`'a koyuyor → PII log sızıntısı (ID: DB-SEC-01)
- **Dosya:** `packages/db/src/errors.ts:28,69`
- **Kanıt (ana-context doğrulandı):** Constructor (`:28`) `super(\`RepositoryError[${cause}]${detail?': '+detail:''}\`)` → detail hem `.detail` hem `.message`'a. `:69` 23505 için `pgErr.detail` geçiyor. PG unique-detail: `Key (email)=(x@y.com) already exists`. `logger.error(err)` (pino default `err.message`) → e-posta/telefon KVKK-PII log'a. `errors.findings.test.ts` KIRMIZI.
- **Etki:** Repo'nun "raw pg sızmaz" sözü ihlali; gerçek sızma api handler loglama şekline bağlı → **Blok 4'te teyit.**
- **Öneri:** PII-kolon unique'lerinde detail redaksiyonu; `.message`'a ham detail koyma (yalnız `.cause`/`.messageKey`). · **Etiket:** MVP-fix

### [HIGH] [BUG] `mergeInto` void'lenmiş ödemeyi "aktif" sayıp birleştirmeyi reddediyor (ID: DB-TX-02)
- **Dosya:** `orders.ts:1577-1586` · **Kanıt (canlı):** ödeme→void (`voided_at` dolu)→`mergeInto` → `RepositoryError[check] ORDER_HAS_PAYMENTS` (`merge-voided-payment-block.findings.test.ts`, KIRMIZI). Kök: 042 (birleştir) 044'ten (void) ÖNCE yazıldı; payment COUNT'una `voided_at IS NULL` filtresi gelmemiş (diğer tüm SUM/COUNT siteleri almış).
- **Senaryo:** Kasiyer yanlış masaya ödeme→void (ADR-033 canlı akış)→doğru masaya birleştir → "ödeme var" ret, oysa aktif ödeme 0.
- **Öneri:** `orders.ts:1581` payment COUNT'una `.where('voided_at','is',null)`. · **Etiket:** MVP-fix (ADR-033+ADR-029 kesişimi)

### [HIGH] [ROB] errors.ts contention/tip PG kodlarını map'lemiyor → ham hata route'a sızıyor (ID: DB-TX-03)
- **Dosya:** `errors.ts:67-85` · **Kanıt:** `mapPgError` 40001/40P01/22P02/22003/25P02 → hepsi `null` (`errors-unmapped-pg-codes.findings.test.ts` 5 KIRMIZI; 40P01 canlı ~1.5sn gerçek deadlock ile üretildi). `default:null` → caller ham `pg.DatabaseError` fırlatır → api `instanceof RepositoryError` eşleşmez → genel 500. Retry/backoff yok.
- **Etki:** 25P02 (DB-TX-05 kökü) + 22P02/22003 (bozuk UUID/int-taşma) gerçekçi; deadlock bugün nadir (dikkatli kilit sırası) ama backstop'suz.
- **Öneri:** 40001/40P01 → `cause:'retryable'` + api'de sınırlı retry; 22P02/22003 → validasyon hatası. · **Etiket:** MVP-fix (map) + Blok-4 (retry katmanı)

### [HIGH] [PERF] `order_items` tablosunda `order_id` index'i yok → en sıcak path'te Seq Scan (ID: DB-TX-04)
- **Dosya:** şema `000_init.sql:280-296,462-466` · **Kanıt (canlı):** pos_test `pg_indexes`: 4 index'in hiçbiri `order_id` ile başlamıyor; `EXPLAIN ... WHERE order_id=? AND tenant_id=?` → **Seq Scan** (`missing-index.findings.test.ts`, KIRMIZI). Repo `WHERE order_id AND tenant_id` desenini her detay/ekle/recalc/iptal/birleştir'de kullanıyor (orders.ts:502,588,855,1591 + payments:296).
- **Etki:** `order_items` append-only (silinmez) büyüdükçe tüm sıcak path'ler lineer yavaşlar; bugün küçük veri → fark edilmiyor.
- **Öneri:** `CREATE INDEX order_items_order_id_idx ON order_items(order_id, tenant_id)` (yeni migration, db-migration-guard). · **Etiket:** MVP-fix

### [HIGH] [ROB] pg.Pool'a `'error'` listener yok → idle bağlantı hatası tüm process'i çökertebilir (ID: DB-TX-06)
- **Dosya:** `connection.ts:18-24` + tek prod tüketici `apps/api/src/index.ts:39` (grep: hiçbir yerde `.on('error')` yok) · **Kanıt:** `pool.listenerCount('error')===0` (`pool-error-handler.findings.test.ts`, KIRMIZI). pg.Pool EventEmitter; idle client ağ hatası + listener yok → Node process crash.
- **Senaryo:** 50 açık masa + Hetzner ağ mikro-kesintisi idle bağlantıyı vurur → tüm API anında çöker (tüm terminaller), PM2 restart eder ama in-flight kayıp + socket'ler düşer.
- **Öneri:** `pool.on('error', err => logger.error(...))` — tek satır. · **Etiket:** MVP-fix
